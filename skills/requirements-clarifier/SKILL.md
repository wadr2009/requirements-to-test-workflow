# 需求澄清器 —— 识别模糊与缺失的需求细节

## 触发条件

- 阶段 1 开始时由编排器调用
- 用户直接要求"找出需求文档中需要澄清的地方"

## 输入

| 参数 | 来源 | 获取方式 |
|------|------|----------|
| 需求文档 | `{OUTPUT_DIR}/00-迭代范围.md` | `read_file` |
| 完整需求（上下文） | `{OUTPUT_DIR}/00-原始需求.md` | `read_file` |
| 设计文档（可选） | 用户指定路径 | `read_file` |

## 执行步骤

### Step 1：读取输入

```python
from hermes_tools import read_file

# 读取当前迭代范围（主要分析对象）
scope = read_file("{OUTPUT_DIR}/00-迭代范围.md")

# 读取完整需求文档（提供上下文，帮助理解功能背景）
full_req = read_file("{OUTPUT_DIR}/00-原始需求.md")

# 如果有设计文档，一并读取
# design = read_file("docs/设计文档.md")
```

### Step 2：逐模块分析需求

对每个功能模块，检查以下维度：

1. **功能边界是否清晰**：
   - 功能的输入/输出是否明确
   - 触发条件是否完整（用户操作？系统自动？定时任务？）
   - 结束条件是否定义（成功标准？失败判定？）

2. **业务规则是否完整**：
   - 数据校验规则（字段类型、长度、格式、取值范围）
   - 状态流转规则（状态机每个转换的触发条件）
   - 权限控制规则（哪些角色可以执行哪些操作）
   - 业务约束（额度限制、时间限制、频率限制等）

3. **异常处理是否定义**：
   - 网络超时/失败的处理
   - 数据不存在的处理
   - 并发冲突的处理
   - 第三方接口失败的处理

4. **交互流程是否清晰**：
   - 页面跳转关系
   - 操作成功/失败的提示
   - 数据回显与刷新时机

5. **数据一致性要求**：
   - 跨表/跨系统的数据同步
   - 事务边界
   - 幂等性要求

### Step 3：生成问题并分类

对每个发现的缺失/模糊点，生成一个具体的、可操作的问题：

**问题格式**：
```
**问题**: [具体问题内容，一句话说清]
**背景**: [为什么这个问题需要澄清，当前文档的缺失点]
**影响**: [不澄清的后果——哪些测试场景无法设计]
**建议**: [可能的答案方向，帮助产品经理快速回答]
```

**优先级分类**：

| 优先级 | 判定标准 | 示例 |
|--------|----------|------|
| **关键** | 影响核心功能流程，不明确则无法设计测试用例 | "审核通过后的状态流转是什么？" |
| **重要** | 影响边缘场景或用户体验，建议澄清 | "超时重试的间隔和次数？" |
| **一般** | 细节问题，可以基于常识推断 | "字段的 maxlength 是多少？" |

### Step 4：写入输出文件

```python
from hermes_tools import write_file

write_file("{OUTPUT_DIR}/01-clarifier-问题清单.md", report_content)
```

## 输出格式

输出文件 `{OUTPUT_DIR}/01-clarifier-问题清单.md`：

```markdown
# 需求澄清问题清单

**文档**: {ITERATION}迭代需求
**分析日期**: [日期]
**问题总数**: X个（关键: Y, 重要: Z, 一般: W）

---

## 模块一：{模块名称}

### 关键问题

1. **问题**: {具体问题内容}
   **背景**: {为什么需要澄清}
   **影响**: {不澄清的后果}
   **建议**: {可能的答案方向}

### 重要问题

2. **问题**: {具体问题内容}
   **背景**: {为什么需要澄清}
   **影响**: {不澄清的后果}

### 一般问题

3. **问题**: {具体问题内容}
   **背景**: {为什么需要澄清}

---

## 模块二：{模块名称}

...

---

## 总结

- 需要立即澄清的关键问题: Y个
- 建议澄清的重要问题: Z个
- 可选澄清的一般问题: W个
- 建议下一步: 运行 requirements-answerer 基于文档尝试回答
```

## 验证步骤

写入后，使用 `read_file` 重新读取并检查：

```python
from hermes_tools import read_file

report = read_file("{OUTPUT_DIR}/01-clarifier-问题清单.md")["content"]

# 验证项：
# 1. 文件中至少有一个关键问题
assert "关键问题" in report, "缺少关键问题分类"

# 2. 每个问题都有"问题"和"背景"两个字段
import re
questions = re.findall(r"\*\*问题\*\*:", report)
backgrounds = re.findall(r"\*\*背景\*\*:", report)
assert len(questions) == len(backgrounds), "问题与背景数量不一致"

# 3. 问题总数与统计一致
# 4. 至少覆盖了需求文档中所有主要模块
```

## 注意事项

- 问题要具体可操作，避免过于宽泛（"系统性能如何？" ❌ → "审核列表页在1000条数据时的加载时间要求？" ✅）
- 不要编造不存在的问题——只问需求文档中真正缺失或模糊的部分
- 不要在 clarifier 阶段回答问题——只生成问题清单
- 如果需求文档已经非常详细，可以输出 0 个关键问题，但仍需检查重要和一般维度
- 优先根据当前迭代范围（`00-迭代范围.md`）分析，历史迭代内容不作为问题来源
