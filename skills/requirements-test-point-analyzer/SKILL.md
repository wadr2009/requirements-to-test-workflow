# 测试点分析器 —— 从需求提取结构化测试点

## 触发条件

- 阶段 3 第一步由编排器调用
- 用户直接要求"分析需求提取测试点"

## 输入

| 参数 | 来源 | 获取方式 |
|------|------|----------|
| 迭代范围 | `{OUTPUT_DIR}/00-迭代范围.md` | `read_file` |
| 模块拆分报告 | `{OUTPUT_DIR}/02-module-splitter-模块拆分.md` | `read_file` |
| 完整需求（上下文） | `{OUTPUT_DIR}/00-原始需求.md` | `read_file` |
| 设计文档（如有） | 用户指定路径 | `read_file` |

## 执行步骤

### Step 1：读取所有输入

```python
from hermes_tools import read_file

scope = read_file("{OUTPUT_DIR}/00-迭代范围.md")["content"]
modules = read_file("{OUTPUT_DIR}/02-module-splitter-模块拆分.md")["content"]
full_req = read_file("{OUTPUT_DIR}/00-原始需求.md")["content"]
```

### Step 2：对每个模块提取测试点

对模块拆分报告中的每个模块，遍历其包含的每个功能点，提取测试点：

**测试点生成规则**：

对每个功能点，从以下维度生成测试点：

1. **正向场景**（至少 1 个）：功能正常执行的主成功路径
2. **反向场景**（至少 1 个）：输入异常时的处理
3. **边界场景**（至少 1 个）：临界值的验证
4. **权限场景**（按角色）：不同角色权限下的行为差异
5. **状态场景**（如适用）：不同前置状态下的行为差异
6. **联动场景**（如适用）：与其他功能的交互

**测试点编号**：`TP001`, `TP002`, ..., `TPnnn`

### Step 3：分析功能依赖

对每个测试点，分析：

1. **前置测试点**：哪些测试点必须先通过
2. **数据依赖**：需要哪些测试数据/前置状态
3. **影响范围**：本功能的变更会影响哪些其他功能

### Step 4：识别影响范围

对每个功能变更，分析：

1. **直接影响**（Direct Impact）：需求变更直接修改了哪些现有功能的逻辑
2. **间接影响**（Indirect Impact）：因数据或状态变更而受影响的上下游功能
3. **回归测试**（Regression）：哪些已有功能需要回归验证

### Step 5：生成测试点详情

对每个测试点生成结构化详情：

```markdown
#### TP001: {测试点描述}

**优先级**: 高 / 中 / 低
**测试类型**: 功能测试 / 接口测试 / 性能测试 / 安全测试
**所属模块**: {模块编号} - {模块名称}
**来源需求**: {需求文档引用}
**前置条件**:
- {条件1}
- {条件2}
**测试数据要求**:
- {数据1}
- {数据2}
**预期结果**:
- {结果1}
- {结果2}
**依赖测试点**: {TP000, TP002} 或 "无"
**覆盖的功能点**: {功能点ID - 功能点名称}
```

### Step 6：写入输出

```python
from hermes_tools import write_file

write_file("{OUTPUT_DIR}/03-test-point-analyzer-测试点.md", report_content)
```

# 1. 验证测试点编号连续
tp_ids = re.findall(r"TP(\d+)", report)
# 检查是否有跳号

fields = ["优先级", "测试类型", "前置条件", "预期结果"]

    # 搜索该 TP 上下文，确认字段存在

# 3. 验证覆盖率

modules_report = read_file("{OUTPUT_DIR}/02-module-splitter-模块拆分.md")["content"]

## 注意事项

- 每个需求功能点**至少有一个对应测试点**
- 测试点编号必须连续（TP001, TP002, ...），不可跳号
- 前置条件和预期结果必须具体，不可空泛（"正常" ❌ → "状态更新为'审核通过'" ✅）
- 影响范围分析要区分直接影响和间接影响
- 对设计文档中的**状态机、接口参数、数据表结构**要给足够的测试点覆盖
