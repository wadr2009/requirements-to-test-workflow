# 需求模块拆分器 —— 按角色和目录拆分为可测试模块

## 触发条件

- 阶段 2 由编排器调用
- 用户直接要求"拆分这个迭代需求"

## 输入

| 参数 | 来源 | 获取方式 |
|------|------|----------|
| 迭代范围 | `{OUTPUT_DIR}/00-迭代范围.md` | `read_file` |
| 审核报告（可选） | `{OUTPUT_DIR}/01-auditor-审核报告.md` | `read_file` |
| 完整需求（上下文） | `{OUTPUT_DIR}/00-原始需求.md` | `read_file` |

## 执行步骤

### Step 1：读取输入

```python
from hermes_tools import read_file

scope = read_file("{OUTPUT_DIR}/00-迭代范围.md")["content"]
audit = read_file("{OUTPUT_DIR}/01-auditor-审核报告.md")["content"]
full_req = read_file("{OUTPUT_DIR}/00-原始需求.md")["content"]
```

### Step 2：提取所有功能点

从迭代范围文档中提取所有独立功能点：

```python
import re

# 从需求文档中提取所有功能描述
# 按段落/列表项拆分功能点
# 每个功能点记录：名称、描述、涉及的用户角色、优先级
```

### Step 3：识别用户角色

从需求文档中识别所有用户角色：

- 从权限描述、操作者限定词（"管理员可以…"、"平台运营…"）中提取
- 从接口文档的角色鉴权中提取（如有）

每个角色的分析维度：

```markdown
- 角色名称
- 权限范围
- 可执行的功能列表
- 可访问的数据范围
- 与其他角色的交互关系
```

### Step 4：构建功能目录树

按"角色 → 业务领域 → 功能组 → 具体功能"的层次组织：

```
规则：
1. 根节点 = 迭代号
2. 第一层 = 用户角色（或客户端类型）
3. 第二层 = 业务领域/菜单组
4. 第三层 = 功能分组
5. 第四层 = 具体功能点 ← 深度 ≤ 4 层
```

### Step 5：分析模块依赖

对每个模块分析：

1. **前置依赖**：本模块功能需要哪些其他模块的功能先完成
2. **数据依赖**：本模块需要哪些其他模块产生的数据
3. **触发关系**：本模块的完成会触发哪些其他模块的操作

生成依赖图（ASCII art）：

```
[模块A] ──数据依赖──▶ [模块B]
    │                      │
    │ 前置依赖             │ 触发
    ▼                      ▼
[模块C] ◀──数据依赖── [模块D]
```

检查循环依赖：
```python
# 如果 A → B → C → A，标记为循环依赖
# 循环依赖是高风险项，需要特别标注
```

### Step 6：风险评估

为每个模块标注风险等级：

| 风险等级 | 判定标准 |
|----------|----------|
| **高** | 涉及资金/核心流程/多模块依赖/第三方接口/状态机复杂 |
| **中** | 涉及重要功能/有外部依赖/有并发场景 |
| **低** | CRUD 操作/展示页面/独立功能 |

### Step 7：写入输出

```python
from hermes_tools import write_file

write_file("{OUTPUT_DIR}/02-module-splitter-模块拆分.md", report_content)
```

- 功能点覆盖率 = 100%（与原始需求对比）
- 依赖关系图无未标记的循环
- 每个模块都有风险等级（高/中/低）
- 至少识别了一个用户角色

## 注意事项

- 按**用户角色**拆分是第一维度，确保每个角色的功能都被独立审视
- 目录树深度 ≤ 4 层，超过则合并相邻层级
- 依赖图中**循环依赖必须有明确标注**，不可被忽略
- 风险等级要基于对业务的影响判定，不是基于实现的复杂程度
- 如果阶段 1 的审核报告中有"需用户确认"项，在相关模块标注"依赖用户确认"
- **并行模式适配**：输出格式中每个功能点带 REQ ID（REQ-001, REQ-002...），模块详情中的功能点列表必须是编排器可解析的结构化格式（每行一个 `REQ-XXX: 描述`），以便编排器在并行模式下提取模块列表并分配命名空间前缀
