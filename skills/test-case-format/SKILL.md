# 测试用例格式规范（XMind 导入标准）

## 核心职责

基于需求文档 + 功能点 + 后端设计生成完整测试用例，覆盖：
- 正常流程
- 异常场景
- 边界值
- 数据校验
- 权限控制
- 状态联动
- 对历史功能的影响

---

## 严格格式规范

### 1. 根节点（一级标题）

- 使用 `# ` 表示，**整个文件只有一个**
- 内容为迭代名称
- 示例：`# 270迭代测试用例`

### 2. 功能模块（二级标题）

- 使用 `## ` 表示
- 内容为功能模块名称
- 示例：`## 一、保理融资审核-单笔审核`

### 3. 功能分组（三级标题）

- 使用 `### ` 表示
- 内容为功能分组名称
- 示例：`### 1.工行开立信息上送`

### 4. 测试用例声明

- 以 `###` 或 `####` 开头（取决于当前层级深度），固定格式：
  - `### tc-P0：测试用例标题`
  - `### tc-P1：测试用例标题`
  - `### tc-P2：测试用例标题`
- 当功能分组层级较深时，用例声明可延伸至 `####` 或更深层级
- **tc- 前缀、P0/P1/P2、中文冒号** 必须严格一致

### 5. 前置条件

- 测试用例的下一级，以 `##### ` 开头，固定格式：
  - `##### pc：前置条件描述`
- 多个前置条件用中文分号分隔
- 示例：`##### pc：用户已登录；账户状态为正常；可用额度 ≥ 1000.00`

### 6. 测试步骤

- 以 `##### ` 开头（与 pc 同级，但无 "pc：" 前缀）
- 直接写操作描述，不加序号
- 示例：`##### 进入签发申请页面，不填写任何字段，直接点击【申请签发】按钮`

### 7. 预期结果

- 以 `###### ` 开头
- **必须紧跟在对应的步骤之后**
- 单行多个结果用中文分号分隔
- 示例：`###### 系统显示所有必填字段的错误提示，包括结算金额、接收方名称、到期日`
- 每个测试用例最多 20 步（即最多 20 对 步骤+预期结果）

---

## 测试用例分级

| 级别 | 定义 | 覆盖范围 |
|------|------|----------|
| **P0** | 冒烟测试用例 | 核心功能、核心流程、关键异常 |
| **P1** | 重要功能用例 | 重要功能、一般异常、边界值、字段校验 |
| **P2** | 次要场景用例 | 次要场景、体验优化、非核心验证 |

---

## 核心生成规则

### 1. 表单字段验证规则
- 同一页面/表单的所有字段验证**合并为一个测试用例**
- 不按字段逐个创建用例
- 在同用例内按字段顺序验证

### 2. 场景法要求
- 使用场景法补充和组织测试用例
- 覆盖：
  - 主成功场景（Happy Path）
  - 基本替代场景（Alternative Paths）
  - 异常场景（Exception Paths）

### 3. 需求覆盖
- 严格对齐原始需求和功能点，确保 100% 覆盖
- 不遗漏、不冗余

### 4. 后端设计对齐
- 结合字段规则、接口逻辑、数据唯一性、状态机、表关系、历史数据兼容性

### 5. 功能联动
- 体现功能间联动及对历史模块的影响

### 6. 可执行性
- 描述简洁、无歧义、可直接执行

### 7. 不确定项处理
- 不明确处标记"需产品/开发确认"，不猜测

## 正确格式示例

```markdown
# 测试模块

## 功能分组

### tc-P1：签发申请表单字段验证

##### pc：用户已登录系统，且有可用授信额度

##### 进入签发申请页面，不填写任何字段，直接点击【申请签发】按钮
###### 系统显示所有必填字段的错误提示，包括结算金额、接收方名称、到期日、接收方地址、签发协议、签发地址

##### 填写结算金额为 0.00，其他字段保持为空，点击【申请签发】按钮
###### 系统显示结算金额的错误提示，提示金额必须大于 0.00

##### 填写结算金额超过可用额度，其他字段保持为空，点击【申请签发】按钮
###### 系统显示结算金额的错误提示，提示金额不能超过可用额度
```

## 可验证的格式检查规则（v2.0 新增）

以下规则可用正则表达式自动验证，test-case-reviewer 必须逐项检查：

### 规则 1：一级标题唯一

```
正则: ^# [^#]
预期: 文件中有且仅有 1 个匹配
```

### 规则 2：用例声明格式

```
正则: ^#{2,5} tc-P[012]：[^\n]+
要求: 所有用例声明匹配此格式
```

### 规则 3：前置条件格式

```
正则: ^##### pc：[^\n]+
要求: 每个用例声明后紧跟至少一个 pc
```

### 规则 4：预期结果格式

```
正则: ^###### [^\n]+
要求: 多结果用中文分号（；）分隔
```

### 规则 5：步骤与结果配对

```
逻辑: 每个 ^##### (?!pc：) 步骤后，必须有对应的 ^###### 预期结果
不允许: 步骤后无结果 / 结果前无步骤
```

### 规则 6：无空步骤

```
不允许: ^#####\s*$ （空步骤行）
```

### 规则 7：用例步骤上限

```
逻辑: 每个用例的 ##### (?!pc：) 数量 ≤ 20
```

### 规则 8：标题层级连续

```
逻辑: # → ## → ### → #### → ##### → ######
不允许: # 直接跳 ####（跳级）
```

### 验证脚本（test-case-reviewer 使用）

```python
import re

def validate_format(file_content: str) -> dict:
    """验证测试用例格式合规性，返回检查结果"""
    
    results = {}
    
    # 规则 1：一级标题唯一
    h1 = re.findall(r"^# [^#]", file_content, re.MULTILINE)
    results["一级标题唯一"] = len(h1) == 1
    
    # 规则 2：用例声明格式
    tc_decls = re.findall(r"^#{2,5} tc-P[012]：[^\n]+", file_content, re.MULTILINE)
    results["用例声明格式正确"] = len(tc_decls) > 0
    
    # 规则 3：前置条件完整
    pc_count = len(re.findall(r"^##### pc：[^\n]+", file_content, re.MULTILINE))
    results["前置条件完整"] = pc_count >= len(tc_decls)
    
    # 规则 4：预期结果格式
    result_lines = re.findall(r"^###### [^\n]+", file_content, re.MULTILINE)
    results["预期结果存在"] = len(result_lines) > 0
    
    # 规则 5：步骤与结果配对
    steps = re.findall(r"^##### (?!pc：)[^\n]+", file_content, re.MULTILINE)
    results["步骤结果配对"] = len(result_lines) >= len(steps)
    
    # 规则 6：无空步骤
    empty_steps = re.findall(r"^#####\s*$", file_content, re.MULTILINE)
    results["无空步骤"] = len(empty_steps) == 0
    
    # 计算合规率
    total = len(results)
    passed = sum(1 for v in results.values() if v)
    results["合规率"] = passed / total * 100
    
    return results
```

## 质量保证要求

- **100% 需求覆盖**：所有给定的需求和功能点必须覆盖
- **不编造**：未提及的功能不创建测试用例
- **实现对齐**：匹配实际后端设计
- **结构正确**：完全合规的 XMind 导入格式
- **可执行**：步骤和结果清晰可操作
- **不确定标记**：未确认的需求标注，绝不猜测
